## 개요

운영 서버 시놀로지 NAS의 MongoDB가 랜섬웨어 공격을 받아 `romrom` DB 전체가 삭제된 사고를 계기로, 자격증명 강화·백업 자동화·test 환경 분리까지 일괄 진행했다. MongoDB/mongo-express 비밀번호를 강력한 값으로 교체하고 mongo-express에 Basic Auth를 걸었으며, PostgreSQL 백업 스크립트와 동일 구조의 `backup_mongo.sh`를 신규 작성해 시놀로지 작업 스케줄러 등록 가능 상태로 만들었다. 추가로 운영(`prod`)과 격리된 `test` 환경(PostgreSQL/MongoDB DB, MinIO 버킷, application-test.yml)을 구축했다.

## 주요 구현 내용

### 랜섬웨어 공격 원인과 해결
공격 표면은 `admin/admin` 기본 자격증명 + 27017 포트 외부 노출 조합. 이는 MongoDB 랜섬웨어 봇이 자동 스캔하는 전형적 패턴이다. "배포할 때마다 데이터가 사라진다"고 보였던 현상의 실체는 **배포 사이사이 봇이 반복 침입**한 것. `chatuserstate` 컬렉션만 살아남은 이유는 컬렉션이 비면 재생성하는 방어 로직이 별도로 존재했기 때문이고, 다른 컬렉션엔 그런 로직이 없어 침입 후 복구되지 않았다.

해결: 자격증명 강화(강력 비밀번호) + mongo-express Basic Auth 활성화. 27017 포트는 외부 클라이언트 접속 요구로 인해 외부 노출 유지, 강한 비밀번호로 봇 무력화.

### `@` 문자가 포함된 비밀번호 처리 — 단계적 시행착오
MongoDB connection URL의 `@`는 호스트 구분자이므로 비밀번호의 `@`가 파싱을 깨뜨린다. 다음 시도들을 거쳤다.

1. `ME_CONFIG_MONGODB_URL` 환경변수에 `%40` 인코딩 → docker-compose가 다시 `@`로 디코딩 → `kimchi:Kimchi123@@mongodb:27017`로 파싱 실패
2. `ME_CONFIG_MONGODB_SERVER` + `ADMINUSERNAME/PASSWORD`만 사용 → mongo-express 1.0.2가 ADMIN 자격증명을 URL에 직접 끼워 넣어 동일한 `@@` 문제
3. `ME_CONFIG_MONGODB_URL=''` 빈값 + `MONGODB_SERVER` 우선 사용 시도 → mongo-express 내부 빌드 로직이 여전히 ADMIN 값을 URL로 조립
4. 외부 config.js 마운트 → `module.exports = {}` (CommonJS) 사용 → mongo-express 1.0.2가 `"type": "module"` 설정이라 `ReferenceError`
5. 외부 config.js를 `export default {}` (ES module) + JS 코드 안에서 `encodeURIComponent('Kimchi123@')`로 직접 조립 → 정상 동작

### MongoDB 사용자 초기화 함정
`MONGO_INITDB_ROOT_USERNAME` 환경변수는 데이터 디렉토리가 **완전히 비어있을 때만** 동작한다. 1차 데이터 삭제 후 컨테이너가 한 번 떠서 빈 상태가 깨진 시점에 새 환경변수가 적용되지 않아 `kimchi` 사용자가 생성되지 않았고, 결과적으로 `MongoServerError: Authentication failed` 발생. 컨테이너 완전 중지 → 데이터 디렉토리 한 번 더 비움 → 재시작으로 사용자 생성 강제했다.

### 백업 검증 최소 크기 조정
`mongodump --archive --gzip`은 빈 DB에서도 메타데이터로 인해 ~300바이트, 1 document에서도 ~430바이트 정도만 출력된다. 초기 검증 임계값 1KB가 너무 빡빡해 정상 백업도 거부됐고, 100바이트로 완화해 해결.

### 시놀로지 sudo 환경 docker PATH 문제
`sudo`로 docker 명령을 실행할 때 PATH에 docker가 등록돼있지 않아 `command not found` 발생. 절대경로(`/var/packages/ContainerManager/target/usr/bin/docker`)로 명시해 회피.

## 주의사항

### 운영 컨테이너 인증 실패 (재배포 필요)
`romrom-back` 컨테이너가 `unhealthy` 상태이며 컨테이너 로그에 `MongoSecurityException: Authentication failed (admin/admin)` 누적 중. 컨테이너 이미지에 박혀있는 application-prod.yml이 옛 자격증명을 사용한다. GitHub Secret `APPLICATION_PROD_YML`은 1시간 전 업데이트됐으나, 이미지 재빌드 + 재배포 사이클이 돌아야 반영됨. 임시 회피 시 `-e SPRING_DATA_MONGODB_URI=...` 환경변수 오버라이드 가능.

### 운영 로그 호스트 마운트 부재
`romrom-back` 컨테이너의 `/mnt/romrom/logs/`가 호스트로 마운트되지 않아(현재 마운트는 `/app`만), 컨테이너 재배포 시 로그 전량 소실. 일별 gz 로테이션은 동작 중이나 컨테이너 안에만 존재. 호스트 볼륨 마운트(`-v /volume1/projects/romrom/logs:/mnt/romrom/logs`) 추가 권장.
