# ❗[버그][Admin] 기존 admin 관련 버그 정리

## 개요

Admin 운영에서 발생하던 3가지 버그를 수정했다. FK 제약 위반으로 물품 삭제가 실패하던 문제, AlertConfig API의 Swagger 문서 누락 문제, 그리고 회원 정지 처리 시 reportId/reportType 검증 순서가 잘못되어 Redis 토큰이 비정상적으로 삭제되던 문제가 대상이다.

## 변경 사항

### 1. AdminItemService — 물품 삭제 cascade 순서 수정

- `RomRom-Application/.../AdminItemService.java`: `deleteItemByAdmin` 메서드 신규 추가
  - **기존 문제**: `AdminApiController`의 `deleteItem`이 `itemService.deleteItemByAdmin`을 직접 호출하여 `ChatRoom → TradeRequestHistory FK 제약`을 위반
  - **수정 내용**: cascade 삭제 순서를 `chat_room Hard Delete → itemService.deleteItemByAdmin → trade_request_history Hard Delete` 순으로 보장
  - **핵심 설계**: `itemService.deleteItemByAdmin` 호출 시점에 TRH가 DB에 존재해야 FCM 수신자(거래 상대방) 수집이 정상 동작 — 따라서 TRH 삭제는 FCM 알림 이후 맨 마지막에 수행

### 2. AdminApiController — 서비스 호출 수정 및 Swagger 문서 추가

- `RomRom-Web/.../AdminApiController.java`
  - `items/delete` 엔드포인트: `itemService.deleteItemByAdmin` → `adminItemService.deleteItemByAdmin` 호출로 변경 (Application 레이어 경유)
  - `items/delete` 엔드포인트: `@ApiChangeLogs` + `@Operation` Swagger 문서 추가
  - `alert-config/get` 엔드포인트: `@ApiChangeLogs` + `@Operation` Swagger 문서 추가
  - `alert-config/update` 엔드포인트: `@ApiChangeLogs` + `@Operation` Swagger 문서 추가

### 3. AdminMemberService — suspendMember 검증 순서 수정

- `RomRom-Application/.../AdminMemberService.java`
  - **기존 문제**: `reportId != null && reportType == null` 검증이 Redis 토큰 삭제(`redisTemplate.delete`) 이후에 위치 → 검증 실패 시 `@Transactional` 롤백으로 DB는 복구되지만 Redis는 롤백 대상이 아니어서 RefreshToken이 영구 삭제됨
  - **수정 내용**: 검증 로직을 메서드 최상단(모든 DB/Redis 변경 이전)으로 이동
  - `log.error` → `log.warn` 변경 (클라이언트 입력 오류는 서버 에러가 아님)

## 주요 구현 내용

### FK 제약 해소 전략

`ChatRoom` 엔티티는 `TradeRequestHistory`에 `@OneToOne(nullable = false)` FK를 가진다. 기존처럼 TRH를 먼저 삭제하면 DB 제약 위반이 발생한다. 해결책으로 Application 레이어(`AdminItemService`)에서 cross-domain 오케스트레이션을 담당하도록 했다. 이미 `AdminTradeService.forceCancel`에서 `chatRoomService.adminForceDeleteChatRoom`을 사용하는 동일 패턴이 존재했으며, 이를 그대로 재사용했다.

### FCM 알림 보존

`itemService.deleteItemByAdmin` 내부에서 TRH 레코드를 조회하여 FCM 수신자(거래 상대방)를 결정한다. TRH를 먼저 삭제하면 수신자 수집 쿼리가 빈 결과를 반환하여 물품 소유자에게만 알림이 전송되고 거래 상대방은 누락된다. cascade 순서를 올바르게 정렬함으로써 기존 FCM 알림 동작을 보존했다.

### Redis 비트랜잭션 특성 대응

Spring `@Transactional`은 JPA(DB) 변경만 롤백한다. Redis 작업(`redisTemplate.delete`)은 트랜잭션 범위 밖이므로 예외 발생 시 복구되지 않는다. 따라서 입력 검증은 반드시 모든 부수 효과(DB write, Redis delete) 이전에 수행해야 한다.

## 주의사항

- `findAllWithMembersByItemId`는 INNER JOIN FETCH를 사용하므로, 채팅방이 없는(PENDING 상태) TRH는 결과에 포함되지 않을 수 있다. 현재 로직에서는 채팅방 유무를 `ifPresent`로 처리하므로 문제없지만, 향후 PENDING 상태 TRH에 대한 추가 처리가 필요하면 별도 쿼리 분리를 고려할 것
- 테스트 코드는 이 프로젝트 정책상 작성하지 않음
