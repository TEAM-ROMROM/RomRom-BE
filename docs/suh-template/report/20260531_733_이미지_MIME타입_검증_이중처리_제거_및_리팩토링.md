# 이미지 MIME 타입 검증 이중 처리 제거 및 리팩토링

## 개요

긴급 수정으로 도입된 `application/octet-stream` MIME 타입 확장자 추론 로직은 정상 반영됐으나, `isValidMimeType`과 `isValidImageMimeType` 이중 검증 구조가 남아있었다. 현재 등록된 MIME 타입이 전부 IMAGE 타입이므로 두 검증이 동일한 결과를 반환하는 중복이었다. 단일 검증으로 통합하고 호출마다 Set을 재생성하던 비효율도 정적 캐싱으로 개선하였다.

## 변경 사항

### MIME 타입 검증 단일화
- `FileUtil.java`: `isValidMimeType` + `isValidImageMimeType` 이중 검증 → `isValidImageMimeType` 단일 검증으로 통합

### MimeType enum 정리
- `MimeType.java`:
  - `MIME_TYPES` Set 제거 (중복)
  - `isValidMimeType` 메서드 제거 (외부 미사용, `isValidImageMimeType`과 동일 결과)
  - `getMimeTypesByUploadType` 제거 → `IMAGE_MIME_TYPES` 정적 캐시 상수로 교체 (호출마다 Stream 재생성 방지)

## 주요 구현 내용

```java
// 변경 전: 호출마다 Stream → Set 재생성
public static boolean isValidImageMimeType(String mimeType) {
    return getMimeTypesByUploadType(IMAGE).contains(mimeType.toLowerCase());
}

// 변경 후: 클래스 로드 시 1회 생성, 이후 O(1) 조회
private static final Set<String> IMAGE_MIME_TYPES = Collections.unmodifiableSet(
    Arrays.stream(MimeType.values())
        .filter(type -> type.getUploadType().equals(IMAGE))
        .map(MimeType::getMimeType)
        .collect(Collectors.toSet())
);

public static boolean isValidImageMimeType(String mimeType) {
    return IMAGE_MIME_TYPES.contains(mimeType.toLowerCase());
}
```

`application/octet-stream` → 파일명 확장자 추론 후 `isValidImageMimeType` 단일 통과로 흐름이 명확해졌다.

## 주의사항

- 향후 이미지 외 다른 파일 타입(문서, 동영상 등) 업로드가 필요해지면 `IMAGE_MIME_TYPES` 방식을 타입별로 분리하거나 `UploadType`별 캐시 Map으로 확장 필요
