# ⚙️[기능추가][Admin] 거래 관리 Admin API

## 개요

`AdminTradeService`를 신규 생성하고 `AdminApiController`에 거래 관련 엔드포인트 4개를 추가하였다.
기존 Admin 컨벤션(단일 `AdminRequest`/`AdminResponse`, `POST` + `multipart/form-data`, 별도 Docs 인터페이스 없음)을 준수하여 거래 목록 조회·상세 조회·강제 취소·강제 완료 기능을 구현하였다.

---

## 변경 사항

### DTO 확장

- `AdminRequest.java`
  - `tradeRequestHistoryId` (UUID): 상세 조회 및 강제 처리 시 식별자
  - `tradeStatus` (TradeStatus): 목록 조회 상태 필터 (null이면 전체)
  - `adminTradeForceReason` (String): 강제 취소/완료 사유 (내부용)

- `AdminResponse.java`
  - `trades` (Page\<TradeRequestHistory\>): 페이지네이션된 거래 이력 목록
  - `tradeDetail` (AdminTradeDetailDto): 거래 상세 응답
  - `AdminTradeDetailDto` inner class: `tradeRequestHistory` + `chatRoom` (nullable)

### 신규 서비스

- `AdminTradeService.java` (`RomRom-Application/service`)
  - `getTradesForAdmin`: 상태/기간/검색어 필터 + 페이지네이션 조회
  - `getTradeDetail`: 양쪽 물품·회원·연결 채팅방 포함 상세 조회
  - `forceCancel`: PENDING/CHATTING/TRADE_COMPLETE_REQUESTED → CANCELED 강제 전환
  - `forceComplete`: CHATTING/TRADE_COMPLETE_REQUESTED → TRADED 강제 전환 + 채팅 시스템 메시지

### Repository 확장

- `TradeRequestHistoryRepository.java`
  - `findTradesForAdmin`: JPQL JOIN FETCH 기반 관리자 목록 조회 쿼리 (상태·기간·키워드 선택 필터)

### API 엔드포인트 추가

- `AdminApiController.java`
  - `POST /api/admin/trades/list` — 거래 목록 조회
  - `POST /api/admin/trades/detail` — 거래 상세 조회
  - `POST /api/admin/trades/force-cancel` — 강제 취소
  - `POST /api/admin/trades/force-complete` — 강제 완료

## 주요 구현 내용

### 강제 취소 (force-cancel)

TRADED·CANCELED 상태를 제외한 모든 거래를 CANCELED로 전환한다.
기존 `cancelTradeRequest` 흐름과 동일하게 상태 변경만 수행하고 채팅 메시지는 발송하지 않는다.

- TRADED → `TRADE_ALREADY_COMPLETED(409)` 예외
- CANCELED → `TRADE_ALREADY_PROCESSED(400)` 예외

### 강제 완료 (force-complete)

CHATTING·TRADE_COMPLETE_REQUESTED 상태의 거래에만 적용된다.
기존 `completeTrade()` 메서드를 재사용하여 양쪽 물품(takeItem/giveItem)을 `EXCHANGED`로 일괄 처리하고,
채팅방이 존재하면 `"운영자에 의해 교환 완료 처리되었습니다."` 시스템 메시지(`TRADE_COMPLETED`)를 전송한다.

- PENDING → `INVALID_REQUEST(400)`: 채팅방 없어 완료 불가
- TRADED → `TRADE_ALREADY_COMPLETED(409)`
- CANCELED → `TRADE_ALREADY_PROCESSED(400)`

### N+1 방지

`findTradesForAdmin` 쿼리에서 `takeItem`, `giveItem`, 각 소유 `member`를 JOIN FETCH하여 N+1을 방지한다.
별도 countQuery를 분리하여 페이지네이션 성능을 보장한다.

## 주의사항

- **JPQL nullable enum 파라미터**: `tradeStatus=null` 조건의 `(:tradeStatus IS NULL OR ...)` 패턴은 Hibernate 6.x 환경에서 런타임 검증이 필요하다. 문제 발생 시 네이티브 SQL Builder 방식(`findItemsForAdmin` 패턴)으로 교체한다.
- **강제 취소 채팅 알림 미발송**: 취소 시 채팅 메시지 없이 상태만 변경된다. 운영 정책에 따라 채팅 알림 추가 여부를 결정할 수 있다.
