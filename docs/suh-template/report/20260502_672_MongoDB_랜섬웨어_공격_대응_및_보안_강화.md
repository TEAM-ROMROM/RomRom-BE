# 🔥❗[긴급][버그][보안] MongoDB 랜섬웨어 공격으로 데이터 삭제 및 보안 강화 필요

## 개요

운영 서버 시놀로지 NAS의 MongoDB가 `admin/admin` 기본 자격증명을 노린 랜섬웨어 공격을 받아 `romrom` 데이터베이스 전체가 삭제되고 비트코인 요구 메시지가 삽입되었다. 오염된 데이터를 모두 폐기하고, MongoDB 자격증명을 강력한 값으로 교체했으며, mongo-express에 Basic Auth를 적용해 동일한 공격이 재발하지 않도록 보안을 강화했다.

## 변경 사항

### MongoDB 데이터 초기화
- `/volume1/docker/mongodb/`: 랜섬웨어로 오염된 모든 데이터 파일 완전 삭제 후 깨끗한 상태에서 신규 컨테이너로 초기화

### Docker Compose 설정 (`/volume1/docker/docker-compose/mongodb/compose.yaml`)
- MongoDB 자격증명 변경: `admin/admin` → `kimchi/********`
- mongo-express 환경변수 제거 후 외부 config.js 파일 마운트 방식으로 전환
- mongo-express가 `kimchi/********` 자격증명으로 MongoDB에 접속하도록 구성

### mongo-express 설정 파일 신규 추가 (`/volume1/docker/docker-compose/mongodb/config/config.js`)
- ES module 형식의 커스텀 config 파일
- `connectionString`을 JS 코드 내에서 `encodeURIComponent()`로 직접 조립하여 비밀번호 특수문자(`@`) 안전 처리
- Basic Auth 활성화: `useBasicAuth: true`, 사용자 `kimchi` / 비밀번호 `********`

## 주요 구현 내용

### 비밀번호 특수문자(`@`) 처리 문제 해결

비밀번호 끝의 `@`가 MongoDB connection URL의 호스트 구분자로 해석되어 `MongoParseError: Protocol and host list are required` 에러가 반복 발생했다. 다음 단계를 거쳐 해결했다.

1. **`ME_CONFIG_MONGODB_URL` 환경변수 방식 실패**: docker-compose가 `%40` 인코딩을 다시 `@`로 디코딩해 `kimchi:********@@mongodb:27017`이 되어 파싱 실패
2. **`ME_CONFIG_MONGODB_SERVER` + `ADMINUSERNAME/PASSWORD` 방식 실패**: mongo-express 1.0.2 내부에서 ADMIN 자격증명을 URL에 직접 끼워 넣어 동일한 `@@` 문제 발생
3. **외부 config.js 마운트 방식 채택**: 컨테이너 내부 `/app/config.js`에 ES module 형식 설정 파일을 마운트, JS 코드 안에서 `encodeURIComponent()`로 비밀번호를 인코딩하여 문제 해결

### mongo-express 1.0.2 ES Module 호환성

mongo-express 1.0.2는 `package.json`에 `"type": "module"`이 설정되어 있어 `/app/config.js`는 반드시 ES module 문법(`export default {}`)이어야 한다. 초기에 작성한 CommonJS 문법(`module.exports = {}`)은 `ReferenceError: module is not defined in ES module scope` 에러로 거부되었다.

### MongoDB 사용자 초기화 문제

`MONGO_INITDB_ROOT_USERNAME` 환경변수는 데이터 디렉토리가 **완전히 비어있을 때만** 동작한다. 1차 데이터 삭제 후 컨테이너가 한 번 떠있던 동안 빈 상태가 아니게 되어 `kimchi` 사용자가 생성되지 않았고, 결과적으로 `MongoServerError: Authentication failed`가 발생했다. 컨테이너를 완전히 내리고 데이터 디렉토리를 한 번 더 비운 뒤 재시작하여 사용자 생성을 강제했다.

### Basic Auth 동작 검증

mongo-express 1.0.2의 `ME_CONFIG_BASICAUTH_USERNAME/PASSWORD` 환경변수는 `ME_CONFIG_BASICAUTH=true`와 함께 설정해야 동작한다. 최종 config.js 방식에서는 `useBasicAuth: true` + `basicAuth: { username, password }` 객체로 명시했다.

## 주의사항

### 보안 권장 사항
- 27017, 8081 포트가 여전히 외부에 공개되어 있다 (사용자 요구사항으로 유지). 추후 클라이언트 접속이 내부망에서만 이루어진다면 `127.0.0.1:27017:27017` 형태로 바인딩 제한 권장
- mongo-express Basic Auth는 HTTPS 위에서만 안전 — Cloudflare 프록시를 통한 HTTPS 종단이 정상 동작하는지 정기 점검 필요

### 데이터 손실
- `likeHistory` 컬렉션, 채팅 메시지 데이터는 백업 없이 전량 손실됨
- 추후 정기 백업 정책 수립 필요 (예: `mongodump` 일일 스냅샷 → 별도 볼륨 보관)

### 무시 가능한 로그
- mongo-express 컨테이너 시작 시 `Waiting for mongo:27017... mongo: Name does not resolve` 메시지가 반복 출력되는데, 이는 Docker 이미지의 `docker-entrypoint.sh`가 호스트명 `mongo`를 하드코딩한 헬스체크 로직 때문이다. 실제 앱은 config.js의 `mongodb` 호스트로 정상 연결되므로 무시 가능

### mongo-express 이미지 deprecation
- `mongo-express:latest`는 Docker Hub에서 deprecated 상태(2024-04 이후 미업데이트). 추후 1.1.x 안정화 시 마이그레이션 검토 필요
