🗒️ 설명
---

Admin 거래 관리 페이지(`/admin/trades`) 진입 시 거래 목록 조회 API가 500 에러로 실패한다.

- 발생 위치: `POST /api/admin/trades/list`
- 현상: 페이지 진입 즉시 목록이 로드되지 않고 `500 Internal Server Error` 반환, 화면에 "거래 목록 로드 실패" 표시
- 응답: `{"errorCode":"INTERNAL_SERVER_ERROR","errorMessage":"서버 내부 오류가 발생했습니다."}`

🔄 재현 방법
---

1. 관리자 계정으로 로그인
2. `/admin/trades` 페이지 진입
3. 검색어 없이 거래 목록이 자동 조회됨 (`searchKeyword=null`)
4. 500 에러 발생, 목록 로드 실패

📸 참고 자료
---

서버 로그 핵심 원인:

```
org.postgresql.util.PSQLException: ERROR: function lower(bytea) does not exist
 Hint: No function matches the given name and argument types. You might need to add explicit type casts.
```

**근본 원인**

- `TradeRequestHistoryRepository.findTradesForAdmin` JPQL이 검색어 조건에서 `LOWER(CONCAT('%', :searchKeyword, '%'))` 형태를 사용한다.
- `searchKeyword`가 `null`(검색어 미입력)일 때, Postgres가 `'%' || ? || '%'`의 바인딩 파라미터 타입을 추론하지 못해 **`bytea`로 기본 처리**한다.
- 그 결과 `lower(bytea)` 함수 시그니처를 찾지 못해 **parse 단계에서 에러** → 500.
- `:searchKeyword IS NULL OR (...)` 가드는 실행을 막을 뿐, Postgres의 **parse-time 타입 해석**을 막지 못한다. (OR 단락 평가 전에 함수 시그니처를 먼저 해석)
- 정상 동작하는 물품 목록 조회(`findItemsForAdmin`)는 native query에서 검색어가 있을 때만 LIKE 절을 동적으로 append하므로 NULL 파라미터가 DB에 도달하지 않아 이 문제가 없다.

관련 파일:

- `RomRom-Domain-Item/src/main/java/com/romrom/item/repository/postgres/TradeRequestHistoryRepository.java` (`findTradesForAdmin`)
- `RomRom-Application/src/main/java/com/romrom/application/service/AdminTradeService.java` (`getTradesForAdmin`)

✅ 예상 동작
---

- `/admin/trades` 진입 시 검색어 없이도 거래 목록이 정상 조회되어야 한다.
- 검색어 입력 시 물품명 / 양쪽 회원 닉네임 대상 LIKE 검색이 동작해야 한다.
- 빈 문자열 검색어는 전체 조회로 처리되어야 한다 (match-all `'%%'` 방지).

**해결 방안**

- JPQL의 `:searchKeyword`를 `CAST(:searchKeyword AS string)`으로 감싸 Postgres에 파라미터 타입을 명시한다 (data + count 쿼리 양쪽).
- 서비스 계층에서 빈 문자열 검색어를 `null`로 정규화하여 전달한다.
