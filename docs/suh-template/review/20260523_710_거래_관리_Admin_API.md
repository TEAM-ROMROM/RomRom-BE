## 🐛 버그 및 로직

### 🚨 **Critical — force-cancel 시 ChatRoom 미처리**
**위치**: `AdminTradeService.java:103-119`

CHATTING / TRADE_COMPLETE_REQUESTED 상태의 거래를 강제 취소할 때
**채팅방이 DB에 그대로 남음**. 사용자가 채팅 목록에서 해당 채팅방을 계속 볼 수 있고
메시지 발송을 시도할 수 있다.

**정상 흐름 비교** (`ChatRoomService.handleRoomExit`):
1. `tradeRequestHistory.changeToCancelIfChatting()` → tradeStatus = CANCELED
2. 상대방 상태에 따라 채팅방 soft/hard delete
3. 상대방에게 시스템 메시지 전송

**현재 forceCancel**:
1. `trade.setTradeStatus(TradeStatus.CANCELED)` → tradeStatus = CANCELED만 변경
2. ChatRoom 삭제 없음, 시스템 메시지 없음

PENDING 상태(채팅방 없음)는 문제 없지만, **CHATTING/TRADE_COMPLETE_REQUESTED 상태**에서는
채팅방이 orphan 상태로 남아 사용자 UX 불일치가 발생한다.

```java
// 현재 (문제 있음):
trade.setTradeStatus(TradeStatus.CANCELED);
return AdminResponse.builder().build();

// 개선 제안:
trade.setTradeStatus(TradeStatus.CANCELED);

// CHATTING/TRADE_COMPLETE_REQUESTED 상태였다면 채팅방도 처리
if (currentStatus == TradeStatus.CHATTING || currentStatus == TradeStatus.TRADE_COMPLETE_REQUESTED) {
    chatRoomRepository.findByTradeRequestHistoryId(trade.getTradeRequestHistoryId())
        .ifPresent(chatRoom -> {
            // 채팅방 삭제 (혹은 운영 정책에 따라 시스템 메시지 후 삭제)
            chatRoomRepository.delete(chatRoom);
            // chatMessageRepository, chatUserStateRepository 도 함께 정리 필요 여부 확인
        });
}
```

**권장**: `ChatRoomService`의 채팅방 삭제 로직(`executeHardDelete`)을 재사용하거나,
별도 `adminForceCancel` 메서드를 `ChatRoomService`에 추가하여 일관성 유지.

### ⚠️ Major — JPQL nullable enum 파라미터 런타임 위험
**위치**: `TradeRequestHistoryRepository.java:134`

```java
"WHERE (:tradeStatus IS NULL OR t.tradeStatus = :tradeStatus) "
```

Hibernate 6.x(Spring Boot 3.x)에서 `tradeStatus`에 `null`을 전달할 때
`(:tradeStatus IS NULL)` 조건의 type inference 없이 처리되는 경우
`QueryException: Unable to determine type of parameter` 가 발생할 수 있다.

`findItemsForAdmin`은 네이티브 SQL Builder 방식으로 enum null 문제를 회피하고 있음.

**검증 필요**: `tradeStatus=null`로 `/api/admin/trades/list` 호출 테스트.

**오류 발생 시 대안**:
```java
// 서비스 레이어에서 null 분기 처리
Page<TradeRequestHistory> tradePage;
if (request.getTradeStatus() == null) {
    tradePage = tradeRequestHistoryRepository.findTradesForAdminAll(
        startDate, endDate, request.getSearchKeyword(), pageable);
} else {
    tradePage = tradeRequestHistoryRepository.findTradesForAdmin(
        request.getTradeStatus(), startDate, endDate, request.getSearchKeyword(), pageable);
}
```
또는 `findItemsForAdmin` 패턴처럼 네이티브 SQL Builder로 교체.

### 💡 Minor — `validateForceCancelable` PENDING 상태 처리 불명확
**위치**: `AdminTradeService.java:157-166`

`validateForceCancelable`에서 PENDING 상태에 대한 검증이 없다.
PENDING은 강제 취소 가능한 상태이므로 통과가 맞지만,
메서드 내에 TRADED/CANCELED만 체크하는 것이 의도적인지 확인 필요.
주석 "PENDING/CHATTING/TRADE_COMPLETE_REQUESTED → CANCELED 가능"이 코드와 일치하므로 이상 없음.
단, 향후 TradeStatus가 추가될 경우 allowlist(허용 목록) 방식이 더 안전하다.

```java
// 현재: 블랙리스트 방식 (TRADED, CANCELED만 막음)
// 권장(선택적): 화이트리스트 방식
private void validateForceCancelable(UUID tradeId, TradeStatus currentStatus) {
    Set<TradeStatus> cancelableStatuses = EnumSet.of(
        TradeStatus.PENDING, TradeStatus.CHATTING, TradeStatus.TRADE_COMPLETE_REQUESTED);
    if (!cancelableStatuses.contains(currentStatus)) {
        // 상태별 에러코드 분기
        if (currentStatus == TradeStatus.TRADED) throw new CustomException(ErrorCode.TRADE_ALREADY_COMPLETED);
        throw new CustomException(ErrorCode.TRADE_ALREADY_PROCESSED);
    }
}
```

### 💡 Minor — `forceCancel` 반환값 불일치
**위치**: `AdminTradeService.java:118`, `AdminApiController.java:358-360`

서비스는 `AdminResponse.builder().build()`를 반환하지만 컨트롤러는 `ResponseEntity<Void>`.
반환값이 사용되지 않으므로 기능 상 문제 없으나, 서비스 시그니처를 `void`로 변경하면
의도가 명확해진다. (`forceComplete`도 동일)

```java
// 현재:
public AdminResponse forceCancel(AdminRequest request) { ... return AdminResponse.builder().build(); }

// 제안:
public void forceCancel(AdminRequest request) { ... }
```

## 📊 리뷰 요약

**전체 평가**: Request Changes (Critical 1건 해결 필수)
**이슈 통계**: Critical 1개 / Major 1개 / Minor 3개

### 핵심 개선 사항

1. **[Critical] force-cancel 시 ChatRoom 미처리**: CHATTING/TRADE_COMPLETE_REQUESTED 상태 거래 강제 취소 시 채팅방이 DB에 남음. 사용자가 취소된 거래의 채팅방을 계속 볼 수 있음. `ChatRoomService`의 채팅방 삭제 로직을 재사용하거나, 채팅방 삭제 후 상태 변경 처리 필요.

2. **[Major] JPQL nullable enum 검증**: `tradeStatus=null`로 목록 조회 시 런타임 오류 여부 반드시 확인. 오류 발생 시 `findItemsForAdmin` 패턴(네이티브 SQL Builder)으로 교체.

3. **[Minor] forceCancel/forceComplete 서비스 반환 타입 정리**: 반환값을 사용하지 않으면 `void`로 변경하여 의도 명확화.

---

> ChatRoom 미처리 이슈는 운영 환경에서 사용자가 취소된 거래의 채팅방을 계속 볼 수 있어
> UX 불일치가 발생한다. Critical로 분류하여 배포 전 수정을 권장한다.
