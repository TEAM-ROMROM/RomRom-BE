## 🐛 버그 및 로직

### ⚠️ Major — JPQL nullable enum 파라미터 런타임 위험
**위치**: `TradeRequestHistoryRepository.java` — `findTradesForAdmin`

```java
"WHERE (:tradeStatus IS NULL OR t.tradeStatus = :tradeStatus) "
```

Hibernate 6.x(Spring Boot 3.x)에서 `tradeStatus`에 `null`을 전달할 때,
`(:tradeStatus IS NULL)` 조건이 type inference 없이 처리되는 경우
`QueryException: Unable to determine type of parameter` 가 발생할 수 있다.

**검증 필요**: null tradeStatus로 `/api/admin/trades/list` 호출 테스트.

**만약 오류 발생 시 대안**:
```java
// 서비스 레이어에서 null 분기
Page<TradeRequestHistory> tradePage = (request.getTradeStatus() == null)
    ? tradeRequestHistoryRepository.findTradesForAdmin(null, startDate, endDate, keyword, pageable)
    : tradeRequestHistoryRepository.findTradesForAdmin(status, startDate, endDate, keyword, pageable);
```
또는 네이티브 SQL Builder 방식(`findItemsForAdmin` 패턴)으로 교체.

### 💡 Minor — `validateForceCancelable` 에러코드 의미 불일치
**위치**: `AdminTradeService.java:162-165`

```java
if (currentStatus == TradeStatus.CANCELED) {
    throw new CustomException(ErrorCode.TRADE_REQUEST_NOT_FOUND); // ← 의미 불일치
}
```

이미 CANCELED된 거래에 `TRADE_REQUEST_NOT_FOUND(404)` 반환은 실제 의미와 다름.
`INVALID_REQUEST(400)` 또는 `TRADE_ALREADY_PROCESSED`가 더 적합하다.

```java
// 제안
if (currentStatus == TradeStatus.CANCELED) {
    log.error("이미 취소된 거래입니다. tradeRequestHistoryId={}", tradeId);
    throw new CustomException(ErrorCode.TRADE_ALREADY_PROCESSED);
}
```

동일하게 `validateForceCompletable`의 CANCELED 케이스도 동일한 수정 권장.

### 핵심 개선 사항

1. **[Major] JPQL nullable enum 파라미터 검증**: `tradeStatus=null`로 목록 조회 호출 시 런타임 오류 없는지 반드시 확인. 오류 발생 시 네이티브 SQL 방식으로 교체 필요.

2. **[Minor] CANCELED 상태 에러코드**: `validateForceCancelable`/`validateForceCompletable`에서 CANCELED 거래에 `TRADE_REQUEST_NOT_FOUND` 대신 `TRADE_ALREADY_PROCESSED` 사용 권장.

3. **[Minor] getTradeDetail 이중 로딩**: 현재 규모에서는 무시해도 되나, chatRoom 조회 쿼리 최적화는 향후 고려 가능.
